

## Core Rust Language Features

To understand how Rust achieves compile-time memory safety, we need to look at the core language mechanisms
that enforce these guarantees.

In the next few slides, we will break down four of the most important concepts in Rust:

---

### Ownership

Ownership is the foundation of Rust’s memory model.

It defines:

* which part of the program owns a resource,
* who is responsible for its lifetime,
* and exactly when that resource should be released.

Instead of relying on a garbage collector or manual `free()` calls scattered throughout the codebase, Rust
enforces ownership rules at compile time.

This allows memory cleanup to remain deterministic while reducing common issues such as double-free and
use-after-free bugs.

---

### Borrowing

Borrowing allows code to access data without taking ownership of it.

Rust supports:

* immutable references for read-only access,
* and mutable references for controlled modification.

The compiler strictly enforces borrowing rules to ensure memory can be shared safely without unnecessary
copying or unsafe aliasing.

This is one of the key mechanisms Rust uses to balance performance with safety.

---

### Lifetimes

Lifetimes are Rust’s compile-time mechanism for validating how long references remain valid.

The compiler tracks relationships between references and the data they point to, ensuring that references
can never outlive their underlying data.

This eliminates entire classes of dangling pointer and invalid reference bugs before the program can run.

---

### Thread Safety

One of the most interesting aspects of Rust is that these same ownership and borrowing rules naturally
extend into concurrency.

The compiler verifies safe access to shared data across threads and prevents many forms of data races at
compile time.

As a result, Rust treats concurrency safety as part of the language model itself rather than relying
entirely on runtime debugging or synchronization discipline.
