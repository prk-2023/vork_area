# Landscape before Rust:

## The Landscape Before Rust

### A Third Programming Model

Before Rust, systems software development generally forced engineers into two very different programming
models.

---

### Model 1: Manual Memory Management

*(C / C++)*

Languages such as C and C++ provide:

* direct hardware access,
* deterministic performance,
* minimal runtime overhead,
* and precise control over memory and execution.

These properties made them the dominant choice for:

* kernels,
* firmware,
* BSPs,
* drivers,
* embedded systems,
* and high-performance infrastructure.

However, the safety model in these languages is largely developer-centric.

Correctness depends heavily on:

* developer discipline,
* careful ownership tracking,
* code reviews,
* testing,
* static analysis,
* and runtime debugging tools.

This provides maximum flexibility and performance, but also makes large systems difficult to reason about
safely over long periods of time.

---

### Model 2: Garbage-Collected Languages

*(Java / Go / Python)*

Languages with automatic memory management approach the problem differently.

Garbage collection simplifies development by automatically reclaiming memory and reducing many classes of
memory errors.

However, these languages typically introduce tradeoffs that are difficult for low-level systems software:

* larger runtime environments,
* less deterministic execution behavior,
* increased memory overhead,
* limited control over memory layout,
* and restricted direct hardware interaction.

For many systems domains — especially kernels, firmware, and real-time environments — those tradeoffs are
often unacceptable.

---

## Rust’s Approach: A Third Programming Model

Rust introduces a fundamentally different approach.

It attempts to combine:

* the performance and control of C/C++,
* with many of the safety guarantees associated with managed languages,
* without requiring a garbage collector.

Rust achieves this through its ownership and borrowing model.

Instead of relying primarily on runtime memory management or developer discipline alone, Rust encodes
memory safety rules directly into the language type system.

The compiler then verifies:

* ownership,
* lifetimes,
* aliasing rules,
* and thread safety

during compilation.

This shifts part of the responsibility for correctness: from the developer alone, to the combination of
language rules and compiler enforcement.

As a result, memory safety becomes a property verified before execution rather than something discovered
later through testing, crashes, or production failures.

That is the core idea behind Rust’s “third programming model”: bringing compile-time safety guarantees into
systems programming while still preserving low-level control and predictable performance.
